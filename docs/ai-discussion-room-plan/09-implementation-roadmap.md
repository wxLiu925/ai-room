# 实施路线

## 阶段 1：规划文档

目标：明确产品边界、架构、领域模型、实时协议、AI 编排和实施顺序。

产物：

- `docs/ai-discussion-room-plan` 下的规划文档。

完成标准：

- 每个核心方向都有独立文档。
- 文档之间不互相矛盾。
- MVP 范围明确。
- 后续实现顺序明确。

## 阶段 2：项目骨架

目标：创建可运行的 Web 项目。

任务：

1. 初始化 Next.js + TypeScript 项目。
2. 配置格式化和基础 lint。
3. 创建基础目录结构。
4. 实现首页和空房间页。
5. 确认本地启动命令可用。

完成标准：

- 本地能启动开发服务。
- 首页可访问。
- 房间页路由可访问。

## 阶段 3：内存版讨论室

目标：不用数据库，先跑通领域流程。

任务：

1. 定义 Room、Participant、Agent、Message、RoomEvent 类型。
2. 实现内存存储。
3. 实现创建房间。
4. 实现添加 AI。
5. 实现发送消息。

完成标准：

- 用户能创建房间。
- 用户能向房间发送消息。
- 页面能展示内存中的消息。

## 阶段 4：Mock AI 讨论

目标：验证多 AI 协作讨论体验。

任务：

1. 实现 Mock Provider。
2. 实现 roundRobin 调度。
3. 实现 AI thinking 和 completed 状态。
4. 将 AI 回复写入消息流。
5. 支持用户继续追问。

完成标准：

- 一个房间可添加至少 3 个 AI。
- 启动讨论后 AI 依次发言。
- 不同 AI 回复风格不同。
- AI 失败不会中断整个讨论。

## 阶段 5：实时同步

目标：让多个浏览器会话共享房间状态。

任务：

1. 接入 Socket.IO。
2. 实现加入房间。
3. 实现房间事件广播。
4. 实现成员状态广播。
5. 实现 `lastSeq` 补事件。

完成标准：

- 两个浏览器打开同一房间能看到同一消息流。
- 一个浏览器发送消息，另一个浏览器实时收到。
- 刷新或重连后可恢复缺失事件。

## 阶段 6：数据库持久化

目标：让房间、消息和事件可恢复。

任务：

1. 接入 PostgreSQL。
2. 建立 Room、Participant、Agent、Message、RoomEvent 表。
3. 将内存存储替换为数据库存储。
4. 查询历史消息。
5. 查询缺失事件。

完成标准：

- 服务重启后房间历史仍存在。
- 页面刷新后能恢复消息。
- 事件序号稳定递增。

## 阶段 7：真实 Provider

目标：在不改变业务流程的情况下接入真实模型。

任务：

1. 实现真实 Provider。
2. 添加 Provider 配置。
3. 添加超时和错误记录。
4. 添加 usage 记录。
5. 添加 mock fallback。

完成标准：

- 房间可选择 mock 或真实 Provider。
- 真实 Provider 失败时不破坏房间状态。
- usage 和耗时可被记录。

## 阶段 8：游戏基础

目标：为狼人杀等玩法建立规则引擎基础。

任务：

1. 定义 GameSession。
2. 定义 GameAction。
3. 定义 GamePlayerState。
4. 实现阶段切换。
5. 实现动作合法性校验。

完成标准：

- 房间可创建游戏会话。
- 游戏阶段由服务端推进。
- 非法动作被拒绝。
- 私有状态不被广播给所有客户端。

## 阶段 9：狼人杀首版

目标：实现最小可玩的狼人杀。

任务：

1. 支持狼人、村民、预言家。
2. 支持身份分配。
3. 支持夜晚行动。
4. 支持白天发言。
5. 支持投票放逐。
6. 支持胜负判断。

完成标准：

- 人类和 AI 可以完成一局简化狼人杀。
- AI 发言通过 Orchestrator 生成。
- 游戏结果由 Rule Engine 判定。

## 阶段 10：质量加固

目标：提升稳定性和可维护性。

任务：

1. 补充领域模型测试。
2. 补充 Provider 测试。
3. 补充实时协议测试。
4. 补充关键端到端测试。
5. 明确成本和并发限制。

完成标准：

- 核心状态推进有测试覆盖。
- Provider 失败路径有测试覆盖。
- 房间事件顺序可验证。
- 文档与实现保持一致。

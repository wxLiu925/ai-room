# 前端计划

## 前端目标

前端应优先支撑房间内的核心协作体验。页面结构必须简单，讨论模式和游戏模式共用房间框架，差异只放在模式面板中。

## 页面结构

### 首页

职责：

- 展示产品入口。
- 创建房间。
- 加入已有房间。
- 选择房间模式。

首版只需要最小表单，不需要复杂营销页。

### 房间配置页

职责：

- 设置房间标题。
- 设置讨论目标。
- 添加 AI 角色。
- 调整 AI 顺序。
- 启动讨论。

配置页可以后续合并到房间页右侧面板，首版根据实现效率决定。

### 房间页

职责：

- 展示房间标题、模式和状态。
- 展示成员列表。
- 展示消息时间线。
- 展示当前发言者和 AI 状态。
- 提供用户输入框。
- 提供启动、暂停、继续等控制。

## 组件边界

### RoomHeader

显示房间标题、模式、状态、在线人数和当前阶段。

### MemberList

显示真人用户和 AI 角色。AI 状态包括空闲、思考中、发言中、失败。

### MessageTimeline

显示用户消息、AI 消息、系统事件、投票事件和游戏行动结果。

### SpeakerStatus

显示当前轮到谁发言，以及哪些 AI 正在思考。

### Composer

用户输入消息。首版只支持文本。

### ControlPanel

提供开始讨论、暂停讨论、继续讨论、进入下一阶段等操作。

### ModePanel

讨论模式显示议题、当前轮次和结论草稿。游戏模式显示阶段、可执行动作、投票状态和私有信息。

## UI 信息架构

推荐布局：

- 左侧：成员列表。
- 中间：消息时间线。
- 右侧：模式面板。
- 底部：输入框和主要操作。

移动端首版只做基本可用：成员列表和模式面板可以折叠。

## 状态展示

AI 状态必须可见但不夸张。

状态类型：

- `idle`
- `thinking`
- `speaking`
- `failed`
- `waiting`

讨论模式重点展示：

- 当前议题。
- 发言顺序。
- 角色观点。
- 讨论结论。

游戏模式重点展示：

- 当前阶段。
- 存活状态。
- 可执行动作。
- 投票进度。
- 该用户或 AI 可见的私有信息。

## 实现顺序

1. 创建首页和房间页静态布局。
2. 用本地 mock 数据渲染成员、消息和状态。
3. 接入 HTTP API 创建和读取房间。
4. 接入 WebSocket 监听房间事件。
5. 根据事件更新消息时间线和成员状态。
6. 接入讨论控制按钮。
7. 添加游戏模式面板的占位结构。

## 约束

- 不做拖拽工作台。
- 不做复杂动画。
- 不做虚拟形象。
- 不在首版展示 AI 内部推理链。
- 不为讨论模式和游戏模式维护两套完全独立 UI。

# 风险与质量

## 技术风险

### 实时状态与持久化不一致

风险：事件已广播但数据库写入失败，导致刷新后状态丢失。

控制：先持久化，再广播。所有房间事件都带递增 `seq`。

### AI 调用慢或失败

风险：真实 Provider 延迟高、失败率不稳定，可能阻塞房间流程。

控制：设置超时、失败事件和 mock fallback。单个 AI 失败不应中断整个讨论。

### 多 AI 上下文混乱

风险：AI 同时发言或引用未完成消息，导致讨论内容错乱。

控制：MVP 只做顺序发言。并行发言后置。

### 游戏私有信息泄露

风险：狼人身份、夜晚行动等私有信息被广播给所有用户或传给无权限 AI。

控制：区分 public state 与 private state。Context Manager 按权限生成上下文。

### Provider 绑定过深

风险：业务代码直接依赖某个模型 SDK，后续难以替换。

控制：业务层只依赖统一 Provider 接口。

## 产品风险

### 范围膨胀

风险：同时做讨论室、游戏、虚拟形象、语音和复杂 Agent，导致首版无法完成。

控制：先完成讨论室 MVP。狼人杀和高级表现层后置。

### 体验不稳定

风险：AI 回复慢、没有状态反馈，用户误以为系统卡住。

控制：明确展示 AI thinking、speaking、failed 状态。

### 游戏规则过早复杂化

风险：完整狼人杀角色和规则会显著拖慢基础平台建设。

控制：先实现通用 GameSession，再实现简化狼人杀。

## 安全风险

### 客户端越权

风险：客户端伪造服务端事件或提交不属于自己房间的命令。

控制：客户端只能提交命令，服务端校验房间成员身份后生成事件。

### AI 越权行动

风险：AI 输出被直接当作系统指令执行。

控制：AI 输出必须经 Orchestrator 和 Rule Engine 校验。

### 成本失控

风险：多 AI、多轮讨论和长上下文导致 API 成本快速上升。

控制：设置每房间、每轮、每 Agent 的预算限制。

## 测试策略

### 单元测试

覆盖：

- Room 状态变化。
- Message 序号生成。
- RoomEvent 序号生成。
- Orchestrator 轮询策略。
- Rule Engine 动作校验。

### 集成测试

覆盖：

- 创建房间到发送消息。
- 启动讨论到 AI 回复。
- Provider 失败到错误事件。
- 重连后按 `lastSeq` 补事件。

### 端到端测试

覆盖：

- 用户创建房间。
- 添加多个 AI。
- 启动讨论。
- 多浏览器同步消息。
- 刷新后恢复历史。

## 质量约束

- 命名必须准确表达业务含义。
- 非必要不新增抽象层。
- 非必要不新增配置项。
- 注释只解释约束和取舍，不复述代码行为。
- 文档必须跟随实现更新。
- 状态修改必须有清晰入口。
- AI 输出不能绕过业务规则。

## 上线前检查

- 房间事件是否全部持久化。
- WebSocket 重连是否可恢复。
- Provider 失败是否可控。
- 私有游戏状态是否不会公共广播。
- API Key 是否只在服务端使用。
- 成本限制是否生效。
- 核心流程是否有测试覆盖。

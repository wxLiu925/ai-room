# 游戏扩展

## 扩展目标

游戏能力应建立在讨论室模型之上，而不是另起一套房间系统。狼人杀是优先扩展方向，但必须等讨论室、实时事件和 AI 编排稳定后再实现。

## 基本原则

- 房间仍是用户和 AI 的共同容器。
- 消息流仍是主要可见交互界面。
- 游戏状态由 Rule Engine 维护。
- AI 只能提交发言和行动意图。
- 私有信息必须由服务端控制可见性。

## 通用游戏模型

### GameSession

表示一个房间中的游戏实例。

核心字段：

- `id`
- `roomId`
- `gameType`
- `phase`
- `status`
- `state`

### GameAction

表示玩家或 AI 提交的游戏动作。

核心字段：

- `id`
- `gameSessionId`
- `actorId`
- `type`
- `targetId`
- `payload`
- `createdAt`

### GamePlayerState

表示参与者在游戏中的状态。

核心字段：

- `participantId`
- `role`
- `alive`
- `publicState`
- `privateState`

## Rule Engine 职责

Rule Engine 是游戏状态权威。

它负责：

- 初始化游戏。
- 分配身份。
- 控制阶段切换。
- 判断动作是否合法。
- 记录投票和行动。
- 计算死亡、放逐和胜负。
- 生成公共事件。
- 生成私有信息。

它不负责：

- 生成 AI 发言文本。
- 调用模型。
- 展示 UI。
- 保存聊天消息的具体样式。

## 狼人杀阶段

狼人杀可按以下阶段逐步实现：

1. 准备阶段：确认玩家和 AI 数量。
2. 发牌阶段：分配身份和阵营。
3. 夜晚阶段：狼人、预言家、女巫等角色行动。
4. 白天阶段：公布夜晚结果。
5. 发言阶段：按顺序发言。
6. 投票阶段：玩家投票放逐。
7. 结算阶段：判断胜负或进入下一晚。

首个游戏版本可减少角色，只保留狼人、村民和预言家。

## AI 与规则解耦

AI 可以输出：

- 发言内容。
- 投票意图。
- 夜晚行动意图。
- 对局势的分析。

Rule Engine 决定：

- 当前是否允许该动作。
- 目标是否合法。
- 动作是否生效。
- 是否产生公开事件。
- 是否产生私有反馈。

示例：AI 说“我投 3 号”不代表投票立即生效。系统必须确认当前处于投票阶段，AI 存活且有投票权，目标也合法。

## 信息可见性

游戏模式必须区分公共状态和私有状态。

公共状态：

- 当前阶段。
- 存活玩家。
- 公开发言。
- 投票结果。
- 死亡公告。

私有状态：

- 玩家身份。
- 狼人队友。
- 预言家查验结果。
- 女巫药水状态。
- 夜晚行动选择。

Context Manager 必须根据 Agent 权限生成上下文，不能把完整 `privateState` 广播或传给无权限 AI。

## 后置能力

- 完整狼人杀角色。
- 上帝视角回放。
- 自定义规则配置。
- 多局积分。
- AI 胜率统计。
- 观战模式。
- 复杂心理画像和长期博弈记忆。
